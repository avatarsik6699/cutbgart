## 0. Контекст и принципы проекта

Три принципа, которым подчинены все решения ниже — держите их перед глазами при любых отклонениях от спеки:

1. **Инференс — только на клиенте.** Сервер никогда не получает и не обрабатывает изображение пользователя. Это не опция, а архитектурный инвариант — от него зависит и приватность, и себестоимость инфраструктуры.
2. **Продукт бесплатный, без аккаунтов, без платежей.** Донат живёт отдельно, на будущей странице портфолио, и не встроен в этот продукт технически.
3. **SEO и производительность — часть функциональных требований**, а не косметика после запуска. Каждое архитектурное решение проверяется на вопрос "не сломает ли это индексацию или Core Web Vitals".

## 1. Обзор продукта

Веб-приложение, позволяющее удалить фон с изображения полностью в браузере пользователя, без загрузки файла на сервер. Модель машинного обучения выполняется через WebGPU (с автоматическим откатом на WebAssembly) прямо на устройстве. Продукт бесплатный и анонимный, без регистрации.

### 1.1 Целевая аудитория
- Основной сегмент: люди, ищущие разовое решение через поиск (портретное фото, товарное фото, фото для документов, лого).
- Вторичный эффект: продукт как SEO-агрегатор трафика и портфолио-кейс автора (не часть данной спецификации напрямую, но влияет на архитектуру маршрутов — см. раздел 6).

### 1.2 Ключевые нефункциональные требования
| Требование | Целевое значение |
|---|---|
| Изображение покидает устройство пользователя | Никогда |
| Время до интерактивности главной страницы (TTI) | < 2.5 c на среднем 4G |
| LCP | < 2.5 c |
| INP | < 200 мс |
| Время первой обработки после загрузки модели | < 2 c на устройстве с WebGPU, < 8 c на WASM-фолбэке |
| Стоимость инфраструктуры при 50k визитов/мес | Не выше стоимости VPS + copeечный CDN-трафик |

## 2. Функциональные требования

### 2.1 MVP — обязательный минимум

**Загрузка изображения:**
- Drag-and-drop зоны на всю рабочую область
- Клик — открытие системного диалога выбора файла
- Вставка изображения из буфера обмена (Ctrl/Cmd+V) — второстепенный приоритет, но дёшево реализовать
- Поддерживаемые входные форматы: JPEG, PNG, WebP
- Лимит размера файла на входе: 20 МБ (жёсткая проверка до начала обработки, с понятным сообщением об ошибке)
- Лимит разрешения: если сторона изображения больше 4096px — клиентский downscale перед подачей в модель (иначе риск OOM на слабых устройствах)

**Обработка:**
- Один файл за раз в MVP (батч-загрузка — в бэклог v2)
- Индикация трёх последовательных состояний: загрузка модели → инференс → готово (см. раздел 7 про состояния)
- Возможность отменить обработку (если она подвисла) и начать заново

**Результат:**
- Просмотр результата с интерактивным слайдером "до / после"
- Скачивание в PNG с альфа-каналом
- Кнопка "обработать другое изображение" — сброс состояния без перезагрузки страницы (модель остаётся в памяти/кэше)

**Обработка ошибок (обязательно для каждого случая):**
- WebGPU недоступен → автоматический, прозрачный для пользователя откат на WASM, с лёгким уведомлением "работаем в облегчённом режиме, будет чуть медленнее"
- Файл превышает лимит размера/разрешения → понятная ошибка с указанием лимита
- Неподдерживаемый формат файла → понятная ошибка
- Сбой загрузки модели (нет сети на моменте первого визита, 404/CORS на CDN) → повторная попытка с кнопкой "попробовать снова"
- Недостаточно памяти устройства во время инференса (OOM) → перехват ошибки, понятное сообщение с советом уменьшить разрешение изображения

### 2.2 Явно вне рамок MVP (backlog v2+)
- Пакетная (batch) обработка нескольких изображений
- Ручная коррекция маски (кисть добавить/убрать зону)
- Замена фона на цвет/картинку/градиент
- Аккаунты, история обработок, облачное хранилище результатов
- Публичный API
- Мобильное приложение

### 2.3 Явно не будет никогда (по продуктовому решению)
- Загрузка изображений на сервер в любой форме
- Реклама на этом домене
- Донат/оплата на этом домене (см. отдельный проект — портфолио)

### 3.1 Обоснование ключевых решений
- **TanStack Start вместо Next.js** — по требованию заказчика; Nitro-бандл идеально ложится на голый VPS без привязки к Vercel.
- **BiRefNet\_lite, а не RMBG-2.0** — RMBG-2.0 распространяется под CC BY-NC 4.0 и требует платной коммерческой лицензии у BRIA без публичного прайса. Базовый BiRefNet — MIT, что даёт близкое к SOTA качество без юридического риска даже при использовании в проекте, косвенно связанном с донатами через отдельный портфолио-сайт.
- **Transformers.js v4** — полностью переписанный WebGPU-рантайм (C++), офлайн-кэш из коробки, урезанный размер бандла — прямое попадание в требования проекта.
- **shadcn/ui на Base UI, а не Radix напрямую** — по состоянию на июль 2026 сам shadcn/ui переключил дефолт на Base UI: команда, изначально создавшая Radix, теперь развивает его "с чистого листа" при постоянной инженерной поддержке MUI, тогда как Radix после покупки WorkOS обновляется заметно медленнее. Модель распространения shadcn/ui (компоненты копируются в ваш репозиторий, а не устанавливаются как чёрный ящик) идеально ложится на принцип "минимум внешних зависимостей, максимум контроля", который мы уже применили к выбору модели и фреймворка.
- **Feature-Sliced Design в облегчённом виде** — `processes` официально исключён из методологии как избыточный слой, а `widgets` для проекта такого размера (несколько страниц, один основной пользовательский сценарий) добавил бы абстракцию без реальной пользы: композиция блоков спокойно живёт на уровне `pages`. Пять слоёв вместо семи — сознательное упрощение, а не недосмотр.

## 4. ML-пайплайн: детальная спецификация

### 4.1 Порядок инициализации при первом визите
1. Страница рендерится и гидрируется без загрузки модели (модель НЕ грузится на маунте компонента).
2. Модель начинает грузиться только по явному действию пользователя — выбору/drop файла. Это защищает LCP/TTI от многомегабайтной загрузки на пустом экране.
3. Параллельно с загрузкой модели — детект возможностей устройства:
   ```
   const hasWebGPU = 'gpu' in navigator && await navigator.gpu.requestAdapter() !== null;
   ```
4. Инференс запускается в **Web Worker**, не в главном потоке — UI обязан оставаться отзывчивым во время обработки.

### 4.2 Выбор `dtype` (квантование)
Transformers.js v4 поддерживает `fp32` (дефолт для WebGPU), `fp16`, `q8` (дефолт для WASM), `q4`. Рекомендация для этого проекта:
- WebGPU-путь: `fp16` — баланс скорости и качества, вдвое меньше трафика на загрузку весов, чем `fp32`.
- WASM-фолбэк: `q8` — стандартный дефолт библиотеки для CPU-инференса, разумный компромисс.

### 4.3 Переключатель "быстрый / максимальное качество"
Заложен в MVP с самого начала, а не отложен в бэклог.

- **Быстрый режим (по умолчанию)** — модель `onnx-community/BiRefNet_lite-ONNX`. Меньше весов на загрузку, быстрее инференс, разумное качество для подавляющего большинства сценариев.
- **Режим максимального качества** — полная `onnx-community/BiRefNet-ONNX`. Тяжелее весы, дольше первая загрузка и сам инференс, но точнее на сложных краях (волосы, мех, полупрозрачные объекты).
- Переключатель — явный UI-контрол (не автоопределение по устройству, хотя устройство можно использовать для *дефолтного* значения: слабые устройства/WASM-фолбэк — открывать в быстром режиме по умолчанию).
- Выбор сохраняется в `localStorage`, применяется при следующем визите без необходимости выбирать заново.
- Смена режима после того, как одна модель уже загружена — требует повторной загрузки весов другой модели; UI обязан явно предупреждать об этом ("переключение на макс. качество — потребуется догрузить ~N МБ").
- Реализуется как отдельная фича на уровне FSD (`features/quality-mode-toggle`, см. раздел 5) — не зашивается напрямую в модуль инференса, а передаётся туда как параметр через публичный API фичи.

### 4.4 Кэширование модели
- `env.useWasmCache = true` — обязательно включить, иначе рантайм-файлы ONNX перекачиваются при каждом визите.
- Дополнительно: собственный Service Worker (`public/sw.js`), кэширующий сами файлы весов модели (`.onnx`) и WASM-бинарники ONNX Runtime по стратегии cache-first с версионированием по хэшу в пути файла.
- Заголовки кэширования на CDN/R2 для файлов модели: `Cache-Control: public, max-age=31536000, immutable` (файл с хэшем в имени, поэтому безопасно кэшировать "навсегда").
- Обе модели (lite и full) кэшируются независимо — переключение режима туда-обратно в рамках одной сессии не требует повторной загрузки уже скачанного варианта.

### 4.5 Постобработка
- Модель возвращает alpha-матовое одноканальное изображение (не бинарную маску) — сохраняет полутона для тонких деталей (волосы, мех, полупрозрачные объекты).
- Композиция маски с оригинальным изображением — через `OffscreenCanvas` в воркере, не в главном потоке.
- Возврат результата в главный поток как `Blob`/`ImageBitmap`, с последующим освобождением памяти (явный `URL.revokeObjectURL` при уходе со страницы/повторной обработке — предотвращение утечек памяти при работе с несколькими изображениями подряд в рамках сессии).

### 4.6 Обработка больших изображений
- До подачи в модель — downscale на клиенте, если сторона > 4096px (см. 2.1).
- Выходное изображение — апскейл маски обратно к исходному разрешению перед композицией, чтобы не терять качество оригинала в финальном PNG.

### 4.7 Кросс-браузерная матрица (обязательна к покрытию тестированием)
| Браузер/устройство | Путь инференса | Приоритет тестирования |
|---|---|---|
| Chrome/Edge desktop (WebGPU) | WebGPU + fp16 | Высокий |
| Safari desktop/iOS (ограниченная поддержка WebGPU) | WASM + q8 фолбэк | Высокий — Safari требует явного тестирования на реальном устройстве |
| Android Chrome | WebGPU (частично, зависит от чипсета) с фолбэком | Средний |
| Старые/слабые устройства | WASM, вероятна деградация по времени | Средний — важно не давать зависать UI |

## 5. Архитектура приложения: Feature-Sliced Design (гибкий режим)

### 5.1 Слои и правило импортов

Пять слоёв, каждый нижестоящий не знает о существовании вышестоящих — это единственное жёсткое правило FSD, и именно его проверяет Steiger:

```
app       → может импортировать: pages, features, entities, shared
pages     → может импортировать: features, entities, shared
features  → может импортировать: entities, shared
entities  → может импортировать: shared
shared    → ничего выше себя не импортирует
```

Слои `processes` и `widgets` сознательно не используются (см. обоснование в разделе 3.1) — композиция блоков страницы, которая в классической семислойной FSD жила бы в `widgets`, здесь просто происходит на уровне `pages`.

### 5.2 Примирение файлового роутинга TanStack Router со слоем `pages`

Важный технический нюанс: у TanStack Router роутинг файловый и завязан на конкретную структуру `routes/`, которая определяет URL. Слой `pages` в FSD — это про *композицию и бизнес-логику страницы*, а не про то, где лежит файл, определяющий URL. Решение — развести эти две вещи:

- `routes/*.tsx` — предельно тонкие файлы: только `loader` (SEO-данные, метатеги через head API) и рендер соответствующей страницы из `pages/`. Ни одной строчки бизнес-логики.
- `pages/*` — сама композиция страницы: какие фичи и сущности собраны вместе, в каком порядке, с какими пропсами.

```
│   │   └── errors.ts                     # типизированные доменные ошибки (раздел 2.1)
│   └── config/
│       └── env.ts                        # типизированный доступ к переменным окружения
│
public/
├── sw.js                                 # service worker кэша весов модели
├── robots.txt
│
scripts/
└── generate-sitemap.ts                   # проход по дереву routes/ → sitemap.xml
```

### 5.4 Публичный API слайсов

Каждый слайс (`upload-image`, `remove-background`, `home` и т.д.) экспортирует наружу ровно то, что описано в его `index.ts` — импорт мимо публичного API (например, `features/remove-background/model/worker` напрямую из `pages/home`) запрещается правилом `fsd/no-public-api-sidestep` в Steiger и ловится на этапе линтинга, а не в код-ревью.

### 5.5 Ключевое следствие для этого проекта
Вся ML-логика инкапсулирована в `features/remove-background` и ничего не знает про роутинг, SEO или конкретную сценарную страницу. Любая новая сценарная SEO-страница — это новый слайс в `pages/`, который просто по-новому компонует уже существующие `features/upload-image`, `features/remove-background`, `features/quality-mode-toggle` и `entities/processed-image`. Добавление десятой SEO-страницы не требует ни одной новой строчки в ML-модуле.

## 7. UI/UX: состояния приложения

Конечный автомат состояний интерфейса (реализовать явно, не через россыпь булевых флагов):

```
idle → model-loading → ready → processing → result
                ↓            ↓         ↓
              error        error     error
```

- **idle** — зона drag-and-drop, ничего не загружено, модель не инициализирована
- **model-loading** — пользователь выбрал файл, идёт первая загрузка весов модели; прогресс-бар с реальным процентом загрузки (Transformers.js v4 отдаёт прогресс download/build/ready — использовать эти колбэки, не имитировать прогресс)
- **ready** — модель загружена и закэширована, инференс ещё не начат (при повторной обработке в рамках сессии это состояние достигается мгновенно)
- **processing** — идёт инференс в воркере; при WASM-фолбэке — явно показать, что это "облегчённый режим"
- **result** — показан слайдер до/после, доступна кнопка скачивания и кнопка "обработать ещё"
- **error** — достижимо из любого состояния, всегда с конкретным сообщением и действием (повтор/сброс), никогда просто "что-то пошло не так"

### 7.1 Доступность (a11y)
- Drag-and-drop зона доступна с клавиатуры (полноценный `<input type="file">` под капотом, не только визуальная зона)
- `aria-live="polite"` регион для объявления смены состояний (загрузка/готово/ошибка) для скринридеров
- Контраст текста, фокус-стейты на всех интерактивных элементах — стандартные WCAG AA требования

### 7.2 Мобильная адаптация
- Drag-and-drop на мобильных заменяется явной кнопкой "выбрать фото" с доступом к камере (`capture` атрибут инпута) — отдельный сценарий использования (сфотографировать прямо сейчас), который стоит поддержать явно.

### 7.3 Переключатель качества в интерфейсе
- Расположение: видим в состоянии **idle**, до выбора файла — пользователь осознанно выбирает режим до старта, а не посреди процесса.
- Две позиции: "Быстро" (дефолт) / "Максимальное качество", с краткой поясняющей подписью разницы (скорость vs точность на сложных краях), не только названием.
- При выборе "Максимальное качество" на слабом устройстве (WASM-фолбэк) — мягкое предупреждение о заметно большем времени ожидания, но без запрета выбора (решение остаётся за пользователем).
- Изменение режима после того, как результат уже получен, доступно из состояния **result** одной кнопкой "пересчитать в максимальном качестве" — частый сценарий: сначала быстро посмотреть результат, затем догрузить точность при необходимости.

## 8. Инфраструктура и деплой

### 8.1 Контейнеры на VPS (docker-compose)
Три сервиса: `nginx`, `app` (Node/Nitro SSR), `umami` + `umami-db` (Postgres). Ключевые настройки:
- Каждый сервис: `restart: unless-stopped`
- Node-контейнер: `init: true` (подключает `tini` как PID 1) для корректной обработки сигналов и отсутствия zombie-процессов
- `umami-db`: volume для персистентности данных Postgres, `healthcheck` перед стартом `umami`

### 8.2 Nginx
- Реверс-прокси на контейнер `app` (порт из Nitro node-server, по умолчанию 3000) и на `umami` (порт 3000 внутри его сети, отдельный поддомен, например `analytics.domain.tld`).
- Gzip/Brotli сжатие на уровне Nginx для текстовых ответов SSR.
- TLS: Certbot с cron-задачей обновления, либо связка `nginx-proxy` + `acme-companion` для автоматизации через лейблы контейнеров.

### 8.3 CDN и хранилище весов модели
- Домен проксируется через Cloudflare (оранжевое облако) — кэширование статики, защита от DDoS, бесплатный TLS на границе.
- Файлы модели (`.onnx`, WASM-рантайм) — не в Docker-образе приложения, а в Cloudflare R2 (нулевая плата за исходящий трафик), отдаются по прямому CDN-URL с хэшем в пути.

### 8.4 CI/CD
1. GitHub Actions на push в `main`: линт → тесты → сборка Docker-образа приложения.
2. Push образа в registry (GitHub Container Registry — бесплатно для публичных репозиториев).
3. SSH-шаг на VPS: `docker compose pull && docker compose up -d`.
4. Отдельный workflow: генерация и загрузка весов модели в R2 запускается только при их изменении (не на каждый деплой кода).

### 8.5 Спецификация VPS
Провайдер: **hip-hosting** — Docker предустановлен на образах по умолчанию, дополнительная настройка не требуется. Поскольку сервер выполняет только SSR лёгкой оболочки страницы (без инференса), достаточно скромной конфигурации: 1-2 vCPU, 1-2 ГБ RAM. Компонент, который реально может вырасти по нагрузке при трафике — это Umami+Postgres, но и они лёгкие при разумном трафике проекта. При росте нагрузки апгрейд тарифа у того же провайдера — самый простой путь, миграция на другого провайдера не требуется architecture-wise (весь стек в Docker, переносим `docker-compose.yml` куда угодно).

## 9. Аналитика и мониторинг

### 9.1 Что и зачем отслеживать
| Метрика | Источник | Зачем |
|---|---|---|
| Визиты, страны, устройства, Core Web Vitals реальных пользователей | Cloudflare Web Analytics | Общая картина трафика и производительности, ноль настройки |
| `model_load_started` / `model_load_completed` / `model_load_failed` | Umami custom event | Понять реальный процент отвала на загрузке модели |
| `processing_started` / `processing_completed` / `processing_failed` | Umami custom event | Ключевая продуктовая метрика — сколько людей реально довели дело до результата |
| `download_clicked` | Umami custom event | Финальная конверсия воронки |
| `webgpu_unavailable_fallback` | Umami custom event | Насколько часто срабатывает WASM-фолбэк — сигнал для приоритизации оптимизации |

Все события — только агрегированные счётчики без PII, без привязки к конкретному изображению или его содержимому (соответствует принципу приватности из раздела 0).

### 9.2 Мониторинг доступности
Uptime Kuma (self-hosted, можно в том же docker-compose) или бесплатный тариф UptimeRobot — пинг главной страницы и `/api/heartbeat` Umami каждые 5 минут, уведомление в Telegram/email при падении.

### 9.3 Логи
Nginx access/error логи — стандартный вывод в stdout контейнера (Docker-native), ротация через `docker-compose` log-driver с ограничением размера, без отдельного лог-агрегатора на старте (избыточно для текущего масштаба).

## 10. Безопасность и приватность

- Никаких серверных эндпоинтов, принимающих файлы изображений — само отсутствие такого API-маршрута исключает целый класс рисков (загрузка вредоносных файлов на сервер, хранение чувствительных данных).
- CSP-заголовки: разрешить загрузку скриптов/WASM только с собственного домена и домена CDN/R2, запретить инлайновые скрипты без nonce.
- Файл модели и веса — публично доступны по дизайну (это open-source веса), скрывать их не требуется и не нужно.
- Политика приватности (простая статичная страница) обязана явно и точно описывать: "изображение не покидает ваше устройство" — это ключевое УТП и должно быть проверяемым утверждением, а не маркетинговой фразой.

## 11. Тестирование

| Тип теста | Что покрывает | Инструмент |
|---|---|---|
| Unit | `features/remove-background` — детект возможностей устройства, обработка ошибок, чистые функции постобработки | Vitest |
| Архитектура | Соответствие проекта слоям FSD, публичный API слайсов, отсутствие кросс-импортов | Steiger (в CI, отдельным шагом до тестов) |
| Integration | Хук `useBackgroundRemoval` с замоканным воркером | Vitest + Testing Library |
| E2E критичного пути | Загрузка файла → обработка → скачивание, на реальном (или headless с WebGPU-флагом) браузере | Playwright |
| Кросс-браузерная матрица | WebGPU-путь и WASM-фолбэк отдельно, обязательно включая Safari/iOS | Playwright projects по браузерам |
| Визуальная регрессия (опционально v2) | UI-компоненты в разных состояниях | Playwright screenshots |

Приоритет: e2e критичного пути и кросс-браузерная матрица важнее unit-покрытия процентами — цена ошибки именно в "не работает на Safari" или "зависает на слабом Android" выше, чем в мелких утилитах.

## 12. Этапы разработки

1. **Каркас** — TanStack Start проект со слоями FSD и настроенным тулингом (ESLint flat config, Prettier, Steiger, Husky), Docker + Nginx + деплой на VPS работающей "hello world" страницы (проверка всей инфраструктурной цепочки до написания продуктовой логики).
2. **ML-ядро** — слайс `features/remove-background` изолированно, с тестовой страницей без дизайна: загрузка модели (обе версии — lite и full), инференс, детект WebGPU/WASM, состояния и ошибки.
3. **Переключатель качества и дизайн-система** — `features/quality-mode-toggle`, установка и настройка shadcn/ui на Base UI, базовый набор компонентов в `shared/ui`.
4. **UI главной страницы** — слайс `pages/home`, полноценная композиция фич поверх готового ML-ядра, все состояния из раздела 7, доступность.
5. **Аналитика** — Umami + Cloudflare Web Analytics, события из раздела 9.
6. **SEO-слой** — сценарные страницы как новые слайсы `pages/*` поверх уже готовых фич, sitemap-генерация, структурированные данные, метатеги.
7. **Кросс-браузерное тестирование и полировка** — обязательный проход по матрице из раздела 4.7 на реальных устройствах, не только эмуляторах.
8. **Запуск** — публикация, без блокировки на готовности портфолио/доната (полностью отдельный трек).
